1

Problem Identification & Motivation

Define the research problem and justify the value of a solution. Understand the problem's relevance, current solutions, and their weaknesses.

Knowledge Base Required
Literature on occupational fraud, limitations of existing audit tools, accounting information systems research.

1.1  The Gap in SME Financial Oversight

Small and medium enterprises (SMEs) suffer disproportionate losses from occupational fraud; ACFE (2022) estimates a median loss of $117,000 per scheme with a median duration of 12 months before detection.
The Financially Responsible Person (FRP) — owner, board treasurer, nonprofit director — is typically a non-accountant who lacks the technical means to monitor bookkeeper activity between audits.
Cloud-based accounting platforms (QuickBooks Online, Wave) have moved transaction data online and made programmatic access feasible, yet no commercially available tool provides automated, interpretable forensic oversight for the FRP.

1.2  Weaknesses of Existing Solutions

ACL (Diligent) and IDEA (CaseWare) require trained auditors and are positioned as post-period analytical tools, not continuous surveillance instruments.
Built-in QBO/Wave reporting lacks statistical anomaly detection, fraud-taxonomy alignment, or a behavioral baseline comparison capability.
Existing anomaly detection research in AIS focuses on large enterprise ERP environments (SAP, Oracle) and requires access to raw SQL databases, unavailable in cloud SME platforms.
No prior AIS artifact addresses the specific combination of: (a) semantic ontology construction from cloud accounting APIs, (b) daily churn profiling, and (c) real-time misappropriation-taxonomy-aligned warnings for a non-expert FRP.

1.3  Research Questions

RQ1: Can daily transaction data from a cloud SME accounting platform be reliably represented as an RDF/OWL semantic snapshot suitable for persistent behavioral profiling?
RQ2: Can a 17-element churn persistence index (phi array) derived from such snapshots provide a statistically defensible baseline for anomaly detection?
RQ3: Can the full pipeline — data extraction, ontology construction, CPI analysis, anomaly identification, misappropriation mapping, and management warning — be implemented as a deployable artifact operable by a non-technical FRP?

1.4  Justification of Value

An FRP-deployable surveillance artifact addresses a documented, economically significant problem that existing tools do not solve for the SME/nonprofit segment.
The artifact sits at the intersection of AIS, semantic web technologies, and LLM-assisted analysis — a convergence that is both timely and novel in the JFAR literature.
Practical relevance is demonstrated by a live field test with a nonprofit organization and a planned university business-school lab deployment.

Define the Objectives of a Solution

Specify the criteria a solution must meet beyond general feasibility and performance. What must the artifact accomplish?

Knowledge Base Required
Knowledge of what is possible and feasible; methods, technologies, and theories that can help define objectives.

2.1  Solution Objectives (O1–O6)

O1 — Semantic Representation: The artifact must extract daily transaction data from QuickBooks Online (via REST/OAuth) or Wave Financial (via GraphQL) and construct an RDF/OWL ontology snapshot (TTL format) that captures the full chart-of-accounts structure and transaction event semantics.
O2 — Churn Profiling: The artifact must compute a reproducible 17-element phi array — the Churn Persistence Index (CPI) — that encodes the proportional activity distribution across recognized transaction types on each operational day.
O3 — Baseline Construction: In Training Mode, the artifact must accumulate daily CPI vectors into a training matrix and derive per-element means and standard deviations sufficient to anchor a z-score-based anomaly threshold.
O4 — Anomaly Detection: In Fraud Analysis Mode, the artifact must compare the current-day CPI against the baseline, rank deviations by z-score magnitude, and identify the top-N anomalous transaction-type dimensions for further analysis.
O5 — Misappropriation Mapping: The artifact must map detected anomalies to recognized asset misappropriation categories (ACFE taxonomy, encoded as Asset_Misappropriation.ttl) and estimate misappropriation risk via LLM-assisted retrieval-augmented generation (RAG).
O6 — FRP-Accessible Warnings: The artifact must produce management-facing warning artifacts (JSON dashboard, human-readable report) that require no accounting or technical expertise to interpret and that are operable by the FRP without developer involvement.

2.2  Design Constraints

The artifact must operate entirely on the FRP's local Surveillance Computer; no server-side infrastructure is required beyond the QBO/Wave cloud APIs.
The artifact must be installable by a systems administrator and subsequently operable by the FRP with no command-line exposure.
The LLM reasoning layer must be deployable locally (Ollama) to preserve financial data confidentiality — no transaction data may be transmitted to external AI services.
The artifact must remain platform-agnostic with respect to accounting source: the Phase I–II pipeline must support both QuickBooks Online and Wave Financial via configuration, not code changes.
3

Design and Development

Create constructs, models, methods, or instantiations in which the research contribution is embedded. How should the problem be solved?

Knowledge Base Required
Application of RDF/OWL semantic modeling, statistical baseline methods, QuickBooks/Wave APIs, Python agent architecture, local LLM (Ollama) RAG.

3.1  Artifact Classification (March & Smith, 1995)

Construct: The 17-element phi array (Churn Persistence Index vector) as a daily behavioral fingerprint of accounting activity.
Model: The two-phase training/fraud-analysis pipeline model; the RDF/OWL ontology schema that maps chart-of-accounts entities to fraud-taxonomy classes.
Method: The z-score-based churn comparison procedure; the RAG-based LLM misappropriation risk assessment procedure.
Instantiation: The working NGAME system implemented in Python, deployable on macOS and Windows, connecting to live QBO or Wave accounts.

3.2  Design Principles and Kernel Theory

Semantic representation principle: RDF/OWL provides a machine-interpretable, standards-based encoding of accounting semantics that enables SPARQL-based querying and alignment with published fraud taxonomies — grounded in the REA Enterprise Ontology literature (Geerts & McCarthy, 2006; Sedbrook & Newmark, 2008).
Interpretable associative metrics principle: The CPI phi array uses proportional transaction-type distributions rather than raw dollar amounts, making the baseline robust to seasonal revenue scale changes and interpretable without domain expertise.
Baseline comparison principle: Z-score anomaly detection against a rolling training matrix operationalizes internal control theory's concept of 'expected vs. actual' at a daily granularity, consistent with continuous auditing research (Murthy & Groomer, 2004).
Separation of concerns principle: The two-phase architecture (Phase I: extraction/ontology; Phase II: analysis/reporting) ensures that data integrity and analytical inference are independently testable and auditable.
Local inference principle: Deploying the LLM layer via Ollama on the Surveillance Computer ensures that raw financial data never leaves the FRP's premises, satisfying confidentiality requirements without sacrificing generative reasoning capability.

3.3  Phase I — Data Extraction and Ontology Construction

Data Extraction Agent: authenticates via OAuth (QBO) or API key (Wave GraphQL), extracts all posted transactions for the current day, and writes a structured JSON intermediary.
Ontology Builder: transforms the JSON intermediary into a today-snapshot TTL file (e.g., quickbooks_ontology_Today.ttl) aligned with the Curated Transaction Types ontology. Simultaneously, yesterday's snapshot is retained for delta comparison.
CPI Analysis Agent: parses both TTL snapshots, computes the 17-element phi array by mapping each transaction event to its canonical transaction-type class and computing proportional frequency.

3.4  Phase II — Analysis, Detection, and Reporting

Training Mode — Matrix Management Agent: appends the day's CPI vector as a new row in NGAME_Training_Matrix.xlsx, recalculates per-element means and standard deviations, and flags when the training window is sufficient for fraud analysis mode (configurable threshold, default N=30 days).
Fraud Analysis Mode — Churn Comparison Agent: retrieves the baseline means/SDs from the training matrix, computes element-wise z-scores for the current-day phi array, and ranks all 17 dimensions by absolute deviation.
Anomaly Identification Agent: selects the top-3 anomalous dimensions and maps each to its corresponding chart-of-accounts line items.
Account Mapping Agent: resolves each anomalous CPI dimension to specific GL accounts in the current ontology snapshot.
LLM Analysis Agent: submits the identified anomalies and GL accounts, along with the Asset_Misappropriation.ttl taxonomy, to the local Ollama LLM via RAG; receives candidate misappropriation risk assessments with rationale.
Management Warning Agent: formats the LLM output and z-score rankings into NGAME_Fraud_Analysis_readable.json and management_dashboard.json — the FRP-facing deliverables.

3.5  Implementation Environment

Language: Python 3.10+; key libraries: rdflib (ontology), openpyxl (training matrix), requests/gql (API clients), ollama (local LLM).
Source control: Git repository (NGAME-POC) with versioned agent scripts, ontology files, and training artifacts.
Dual-mode entry point: ngame_dual_mode.py auto-selects Training or Fraud Analysis based on training matrix row count.
Web UI: Flask-based management dashboard (ngame_ui/) providing the FRP with a browser interface requiring no command-line interaction.
4

Demonstration

Prove that the artifact works by solving one or more instances of the problem. Demonstrate the use of the artifact.

Knowledge Base Required
Knowledge of how to use the artifact to solve the problem; knowledge of QBO sandbox and live account environments.

4.1  Demonstration Contexts

Sandbox Validation: NGAME was executed against an Intuit-provided QBO sandbox company (Craig's Design and Landscaping) to confirm end-to-end pipeline execution: extraction → ontology construction → CPI computation → training matrix accumulation → anomaly detection → management warning output.
Anomaly Injection Testing: A dedicated Anomaly Injection Agent (ngame_anomaly_injection_agent.py) was developed to systematically inject known deviations into the transaction stream. NGAME's detection outputs were compared against injected patterns to confirm that the artifact identifies the correct CPI dimensions.
Non-Profit Field Test: NGAME was deployed against the live QBO account of a consenting nonprofit organization, with the developer acting as the FRP. Training phase was completed over a 30-day window; fraud analysis mode was activated and produced management warnings on live data.

4.2  Illustrative Run — Anomaly Injection Scenario

Scenario: A single-day spike in 'Other Expenses' transactions (a common concealment vector for asset misappropriation) was injected at 3× the baseline proportion.
NGAME response: The CPI phi array registered a z-score of 4.1 on the 'Other Expenses' dimension (well above the default alert threshold of |z| > 2.5). The anomaly was ranked #1, mapped to GL account 'Miscellaneous Expense,' and the LLM analysis agent returned a risk assessment citing the ACFE 'Fictitious Vendor' and 'Billing Scheme' categories.
Output artifact: The management dashboard flagged the anomaly with plain-language text: "Unusual spike in Miscellaneous Expense activity on [date]. Recommend immediate review of vendor invoices for this period."
This instance demonstrates the complete pipeline from detection through FRP-interpretable warning.

4.3  University Business School Lab (Planned Demonstration)

A multi-company academic deployment is planned at a university School of Business, in which students construct fictitious nonprofit QBO companies, play alternating bookkeeper/FRP roles, and use NGAME to detect peer-introduced anomalies.
This deployment will provide a structured, reproducible demonstration context with controlled anomaly scenarios and documented detection outcomes.
5

Evaluation

Observe and measure how well the artifact supports a solution to the problem by comparing the objectives against observed results. How well does the artifact work?

Knowledge Base Required
Knowledge of relevant metrics: precision, recall, false-positive rate; evaluation techniques: testing, observational, descriptive.

5.1  Evaluation Strategy

Technical evaluation (Hevner et al., 2004): Does the artifact execute correctly and reliably? — Verified via unit tests on agent modules, end-to-end integration tests, and anomaly injection test suite (run_anomaly_injection_test.py).
Design evaluation: Does the artifact embody the stated design principles? — Assessed by reviewing the ontology schema against REA/AIS literature and confirming that the CPI phi array preserves interpretability across scale changes.
Purpose evaluation: Does the artifact solve the problem as defined? — Assessed by comparing NGAME's top-3 anomaly rankings against ground-truth injected anomalies (precision and recall).
Instrument evaluation: Are the z-score thresholds and training window parameters appropriately calibrated? — Sensitivity analysis across threshold values (|z| = 2.0, 2.5, 3.0) and training window sizes (N = 20, 30, 60 days).
Generalization evaluation: Does the artifact perform consistently across different accounting environments (QBO vs. Wave; for-profit vs. nonprofit chart-of-accounts structures)?

5.2  Preliminary Results from Anomaly Injection Suite

Across a set of 12 injected anomaly scenarios (spanning billing schemes, expense manipulation, payroll anomalies, and vendor fraud patterns), NGAME correctly ranked the injected dimension in the top-3 anomalies in 10 of 12 cases (83% top-3 recall).
False positive rate at |z| > 2.5 threshold: 1.2 false alerts per 30-day baseline window on clean transaction streams — consistent with the statistical expectation for a z-score threshold at 2.5 standard deviations.
These results confirm proof-of-concept detection capability and provide a baseline for further calibration.

5.3  Rigor and Relevance (Hevner et al., 2004)

Relevance: The artifact addresses a documented, economically significant problem for SMEs and nonprofits; it is deployed on live accounting data; its outputs are validated by a domain-knowledgeable FRP (the developer acting as consultant).
Rigor: The design is grounded in established AIS theory (REA ontology, continuous auditing literature), semantic web standards (RDF/OWL), and statistical anomaly detection methods. The knowledge base at each DSRM stage is explicitly documented.

5.4  Limitations

The current evaluation relies primarily on controlled anomaly injection; field evaluation on genuinely fraudulent transaction streams is not yet available and is a planned future study.
The 17-element phi array taxonomy is derived from QBO/Wave transaction-type categories and may require adaptation for other accounting platforms.
LLM output quality depends on Ollama model version and local compute; output variability across model versions has not been systematically evaluated.
The training window assumption (N ≥ 30 days of normal behavior) requires that the baseline period precede any fraud initiation — a condition that cannot be guaranteed in practice but is standard in behavioral baseline literature.

Communication

Knowledge Base Required

6.1  Positioning in the AIS Literature

NGAME extends the AIS design science tradition that spans McCarthy (1982) through Murthy & Groomer (2004), Bovee et al. (2005), and Sedbrook & Newmark (2008) — all of which apply emerging technologies to solve defined AIS problems.
NGAME repurposes the REA ontology tradition (Geerts & McCarthy, 2006) toward a surveillance use case rather than an enterprise system design use case, constituting a 'repurposes' link in the artifact network (Geerts, 2011).
The integration of a local LLM reasoning layer into an AIS artifact is novel in the JFAR literature and speaks directly to the journal's interest in emerging technologies applied to accounting information systems.

6.2  Contribution Statement

This paper contributes: (1) a new instantiation — the NGAME system — that makes continuous behavioral surveillance of SME accounting data feasible for non-expert FRPs; (2) a new method — the Churn Persistence Index (phi array) — as a fraud-sensitive daily behavioral fingerprint for cloud accounting data; (3) a demonstration that local LLM (RAG) reasoning can be integrated into an AIS artifact in a way that satisfies financial data confidentiality requirements.
The contribution is classified as an 'Improvement' in Gregor & Hevner's (2013) knowledge contribution matrix: a new solution to a known problem in a known application domain, with generalizability to other SME cloud accounting platforms.
