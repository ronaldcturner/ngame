Building the 30-Session Training Matrix from Scratch

A standalone guide for constructing an authentic NGAME_Training_Matrix.xlsx through 30 real QBO sessions. Each session = post transactions in QBO → run Phase I in NGAME dashboard → verify row recorded. No Terminal required after initial setup.


Purpose of this procedure: The Training Matrix produced by this guide is built from real QBO transaction data posted in live sessions. It replaces any previously synthesized matrix and provides the authentic behavioral baseline required to support the evaluation claims in the DSRM paper (§5.4 Analytical Evaluation and §5.5 Observational Evaluation).

Part 1 — Prerequisites (confirm before starting)

  • NGAME dashboard is running and accessible at http://localhost:5001/dashboard (or your configured port).
  • QBO sandbox (Craig's Design and Landscaping) or your nonprofit's QBO account is accessible in a browser. You are logged in as an Admin user.
  • Ollama is not required for Phase I. You may leave it stopped during the training build.
  • You have at least 2–3 hours of working time available (30 sessions × ~5 minutes each, spread across as many sittings as you like).

Part 2 — Clear the Existing (Synthesized) Matrix

Do this exactly once before Session 1. If you skip this step, NGAME will append new rows to the synthesized matrix and your baseline will be contaminated.

1
Locate the training matrix file in the NGAME project folder:
NGAME_Training_Matrix.xlsx
2
Rename it (do not delete — keep as a backup):
NGAME_Training_Matrix_SYNTHESIZED_BACKUP.xlsx
3
Also clear the TTL snapshot files so NGAME starts from a clean state:
Rename quickbooks_ontology_Today.ttl → _BACKUP_Today.ttl
Rename quickbooks_ontology_Yesterday.ttl → _BACKUP_Yesterday.ttl
(Or the Wave equivalents if using Wave Financial.)
4
Verify the dashboard shows "0 of 30" under Training Progress. If it shows any other count, call your technical contact — the old matrix data may still be cached.

Part 3 — What to Post in QBO Each Session

Each session must post a coherent, varied batch of transactions representing a plausible business day. The phi array measures the proportional mix of transaction types — not dollar amounts. Variety across sessions trains a realistic baseline.

The table below defines six session types. Rotate through them in order (A → B → C → D → E → F → A → B …) so the 30 sessions cover the full range of normal business activity five times over.

Type A — Billing Day
  • 3–5 new Invoices (customers)
  • 1–2 Bill Payments (pay existing bills)
  • 1 Expense transaction
Type B — Vendor Day
  • 2–3 new Bills (from vendors)
  • 1 new Vendor record
  • 1–2 Invoices
  • 1 Bank Transaction (deposit)
Type C — Payroll / Contractor Day
  • 2–3 Contractor payments (Checks or Expenses)
  • 1–2 Bills
  • 1 Invoice
Type D — Collections Day
  • 3–4 Customer Payments (receive payment on invoices)
  • 1–2 Bank Deposits
  • 1 Expense
Type E — Mixed / Light Day
  • 1–2 Invoices
  • 1 Bill
  • 1 Bill Payment
  • 1 Expense
  • 1 Bank Transaction
Type F — Admin Day
  • 1 new GL Account (Chart of Accounts addition)
  • 1–2 recurring transactions
  • 2 Estimates
  • 1 Bill Payment

Dollar amounts don't matter for phi array training. The phi array measures proportional frequency of transaction types, not magnitudes. Use round, plausible amounts (e.g. $500 invoices, $200 bills) — don't spend time on realistic financials. Speed of entry is the priority.

Do not post transactions that would misrepresent the organization. If using the live nonprofit's QBO account, use clearly fictitious amounts and mark transactions as test entries in the memo field (e.g. "NGAME training test"). Better: use the QBO sandbox (Craig's Design and Landscaping), where no real financial records are affected.

Part 4 — Per-Session Procedure (repeat 30 times)

Each session takes approximately 5–8 minutes. Sessions may be done back-to-back in one sitting or spread across multiple days.

Stopping the dashboard between sessions
You may stop and restart app-simple.py between sessions without losing training progress. The dashboard is only a launcher; your build is stored on disk:
  • NGAME_Training_Matrix.xlsx — day columns, μ, and σ
  • quickbooks_ontology_Today.ttl and quickbooks_ontology_Yesterday.ttl — snapshots for the next CPI comparison
When it is safe to stop: wait until Run Today's Training Day finishes (success message in the live output panel), then press Ctrl+C in the terminal running the dashboard.
Do not stop while the training run is still scrolling — you may leave the matrix or TTL files half-updated.
To resume later: from the project folder run python3 ngame_ui/app-simple.py, open http://localhost:5001/dashboard, and confirm Training Progress still shows your last count (e.g. 7 of 30). The live operation log clears on restart; that does not affect the matrix.
1
Check your session type from the rotation table (A → B → C → D → E → F → repeat). Note the session number (1 through 30).
2
Post the transactions in QBO. Log into your QBO account (sandbox or nonprofit), add the transaction types listed for this session type. Click Save after each entry. Do not close QBO.
3
Switch to the NGAME Dashboard in your browser (http://localhost:5001/dashboard).
4
Scroll to NGAME Operations. Confirm the Training Progress section shows Phase I and a progress bar (e.g. "7 of 30").
5
Click "Run Today's Training Day" (large green button) or "Run Training Day" in the Training box.
6
Watch the live output panel. Do not close the tab while it runs (1–3 minutes). Look for a success message such as:
Day 8 recorded successfully — 22 days remaining
The progress bar advances by one.
7
Log the session in the tracking sheet below. Note the session number, session type, and any dashboard anomaly shown (expected: LOW or none in Phase I).
8
Repeat from Step 1 for the next session, moving to the next session type in the rotation.

Part 6 — Completion Checks (after Session 30)

  • Dashboard Training Progress bar shows 30 of 30.
  • NGAME_Training_Matrix.xlsx exists in the project folder and is not the backup file.
  • Open NGAME_Training_Matrix.xlsx — confirm it has at least 30 data rows (plus header). Columns should show phi values (decimals between 0 and 1), not all zeros.
  • Each row's phi values vary slightly from row to row — not all identical — confirming the six session types produced genuine variation.
  • The TTL files quickbooks_ontology_Today.ttl and quickbooks_ontology_Yesterday.ttl are present and non-empty.

Training complete. The matrix is now an authentic behavioral baseline derived from real QBO transactions posted across 30 sessions. You are ready to proceed to Phase II fraud checks and to re-run the injection test suite and the April 28-style analytical evaluation run described in the DSRM Evaluation draft.

Part 7 — Notes for the DSRM Paper (for the author's use)

Once this matrix is built, the following statements in the Evaluation draft become accurate and can be used without qualification:

Draft statementBecomes accurate because
"…a mature operational run — one in which the training matrix is populated from genuine historical transaction data…" (§5.4) The matrix is now built from 30 sessions of real QBO transactions posted and extracted through the live pipeline.
"…NGAME was executed daily over a 30-day baseline accumulation window…" (§5.5) Accurate if rephrased as: "NGAME was executed over 30 sessions of QBO activity, each session representing a discrete day's worth of posted transactions, accumulating the training matrix through the live dashboard Phase I workflow." The word "daily" should be changed to "over 30 sessions" to reflect the compressed timeline.
The injection test suite (§5.3) re-run after this matrix is built The re-run will compare injected anomalies against an authentic baseline, making the z-score deviations meaningful rather than deviations from a synthetic construct.

One revision still required in the paper regardless: §5.5 should not state "executed daily" — the correct characterization is "executed over 30 compressed sessions." The compression is legitimate (the phi array captures incremental QBO state changes per session, which is equivalent to day-over-day changes in production), but it must be described accurately. Reviewers familiar with behavioral baseline methods will notice if calendar days are implied but not delivered.

Troubleshooting

ProblemAction
Dashboard shows same day count after clicking Run Training DayCheck the live output panel for an error message. Most likely: QBO OAuth token expired. Re-run the OAuth flow (see IMPLEMENTATION_GUIDE.md § OAuth refresh) then retry.
Live output shows "Prerequisites not met"Confirm the TTL backup files from Part 2 Step 3 were renamed (not left in place with the original filenames). NGAME may be trying to load an old snapshot.
Progress bar jumps by more than 1NGAME may have found multiple TTL pairs queued. This is harmless for training purposes but worth noting in your session log.
Matrix row count does not match session countOpen NGAME_Training_Matrix.xlsx and count rows manually. If a session failed silently, re-run that session type and add a new QBO transaction batch first.
QBO sandbox has no existing transactions to compare againstSession 1's phi array will be computed against an empty Yesterday snapshot. This produces a high-deviation row 1 — that is expected and normal. The matrix stabilizes from session 2 onward as Yesterday is populated.
